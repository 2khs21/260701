1. 수행한 작업 (What I Did)
정확도 향상과 토큰 효율성 극대화를 위해 엔드 투 엔드 RAG 파이프라인을 설계하고 구축했습니다

인덱싱 고도화 : 원본 PDF에서 페이지 번호, 머리글, 도트 리더(목차 점) 등 노이즈를 정제하고, 단순 글자 수가 아닌 §(조항) 경계를 기준으로 청킹을 수행했습니다
하이브리드 검색 엔진: **벡터 검색(의미 유사도)**과 **BM25(키워드 매칭)**를 결합하고, RRF(Reciprocal Rank Fusion)와 메타데이터 기반 재랭킹을 적용했습니다
쿼리 및 컨텍스트 최적화: 자연어 질문에서 암시적 조항 번호를 추출해 쿼리를 보강하고, 검색에서 누락된 조항을 강제로 주입하거나 본문 핵심부만 발췌하는 기능을 구현했습니다
생성 및 모니터링: Claude API를 연동하여 적응형 토큰 제한을 적용하고, 답변의 근거가 된 인용(Citation) 추출 및 검색 품질 경고 시스템을 UI에 반영했습니다


2. 설계 사고 과정 (Thought Process)
RAG 시스템의 근본적인 딜레마인 **'입력 과잉(노이즈)'**과 '과도한 필터링(컨텍스트 누락)' 사이의 균형을 잡는 데 집중했습니다

구조적 우선순위: 법규(CFR) 텍스트의 특성상 조항 번호와 체계가 중요하므로, 시맨틱 벡터에만 의존하지 않고 법규의 구조적 데이터를 최대한 활용하는 방식을 택했습니다
비용과 성능의 트레이드오프 돌파: 무조건 모든 문서를 LLM에 넣는 대신, 질문의 복잡도에 따라 컨텍스트를 동적으로 확장하거나 축소하여 비용(토큰)은 줄이고 품질은 유지하고자 했습니다
사용자 신뢰 확보: 모델이 환각을 일으키지 않도록 엄격한 시스템 프롬프트를 설계하고, 검색 결과가 낮을 때 사용자에게 투명하게 경고를 알리는 방식을 도입했습니다


3. 해결한 문제 (Problems Solved)

검색 정확도 저하: 조항 번호(예: §91.131)와 같은 고유 명칭이 벡터 검색에서 무시되는 문제를 BM25 보강 및 조항 힌트 주입으로 해결했습니다
컨텍스트 단절: 조항이 여러 청크로 쪼개져 헤더 정보가 유실되는 문제를 **'앵커 마그넷 프로세스'**를 통한 조항 단위 청크 병합으로 극복했습니다
토큰 낭비: 목차나 무의미한 표 데이터가 LLM에 주입되는 것을 인덱싱 단계의 필터링과 키워드 발췌 기법으로 차단하여 입력 컨텍스트를 최대 30~50% 축소했습니다
환각 및 근거 부재: 검색 결과가 없을 때도 답변을 지어내는 문제를 Zero-Hit 차단 기능과 엄격한 인용 규칙을 통해 원천적으로 방지했습니다


4. 얻은 결과 (Results Obtained)

효율성: 단순 사실 확인 질문의 경우 입력 컨텍스트를 대폭 축소하여 토큰 비용을 획기적으로 절약했습니다
정확도: 복잡한 비교 및 열거형 질문(공역 등급 비교 등)에서 관련 조항들을 라운드로빈 방식으로 골고루 배치하여 답변의 완결성을 높였습니다
투명성 및 UX: 사용자가 검색의 신뢰도를 판단할 수 있도록 경고 배지를 제공하고, 실제 사용된 원문 조항을 즉시 확인할 수 있는 인용 UI를 구축했습니다
안정성: 검색 품질이 낮더라도 무조건 스킵하는 대신 경고와 함께 답변을 시도하여 사용성(UX)을 개선하는 동시에 신뢰도를 유지했습니다
